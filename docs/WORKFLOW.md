# Workflow Res-CRM — обязательный порядок работы

Все изменения в коде (новая функция, багфикс, рефакторинг) должны проходить через этот воркфлоу. **Тесты обязательны** на всё, что затрагивает поведение.

---

## 1. Классификация задачи

Перед началом определи тип задачи:

| Тип | Описание | Роли по порядку |
|-----|----------|------------------|
| **Новая функция** | Новый API, страница, бизнес-логика | Architect → Implementer → Tester → Guardian → Documenter |
| **Багфикс** | Исправление ошибки без изменения контрактов | Debugger → Tester → Guardian → (Documenter при необходимости) |
| **Рефакторинг** | Улучшение кода без изменения поведения | Analyzer → Refactorer → Tester → Guardian → (Documenter при необходимости) |
| **Структурное изменение** | Миграции БД, смена архитектуры | Architect → Implementer → Tester → Guardian → Documenter |

---

## 2. Этапы и обязательные выходы

### 2.1 Architect (архитектор)

**Когда:** новая функция, структурное изменение.

**Обязательный выход:**
- План в `docs/plans/YYYY-MM-DD-<slug>.md`:
  - Цель и границы изменения
  - Затрагиваемые модули (backend/frontend/API)
  - Риски и миграции (в т.ч. БД)
  - План отката (rollback)
  - Критерии приёмки и что должно быть покрыто тестами

**Правила:**
- Не менять структуру коллекций/таблиц без плана миграции.
- Не ломать обратную совместимость API без явного решения в плане.

---

### 2.2 Implementer / Debugger / Refactorer (исполнитель)

**Когда:** после плана (для новой функции/структуры) или сразу (для багфикса/рефакторинга).

**Обязательный выход:**
- Краткий change-log (в коммите или в `docs/plans/...`):
  - Что изменено (файлы, эндпоинты, компоненты)
  - Почему
  - Как проверить вручную (шаги)

**Правила:**
- Зависимости через DI (`get_db`, `get_current_user`, контекст), без новых глобальных синглтонов.
- Явная обработка ошибок, без «голых» `except`.
- Логирование без секретов.

---

### 2.3 Tester (тестировщик) — обязательный этап

**Когда:** после реализации (или параллельно с ней по TDD). **Ни одна фича/багфикс не считаются готовыми без тестов.**

**Обязательный выход:**
- **Backend:** тесты в `backend/tests/`:
  - Юнит-тесты для новой/изменённой логики (`app/services/`, `app/models/`).
  - API-тесты для новых/изменённых эндпоинтов (happy path, 401/403, 422, 404).
  - Регрессия: существующие тесты зелёные.
- **Frontend:** (по мере внедрения):
  - Юнит/компонентные тесты для новой логики (при наличии Vitest/Jest).
  - Сейчас: явный чек-лист ручной проверки в change-log или в плане.
- Краткий отчёт: что покрыто, как запустить (`pytest ...`, сценарии E2E).

**Правила:**
- Новый API → минимум один тест на успех и один на ошибку (валидация/авторизация).
- Новая бизнес-логика в `calc.py` или сервисах → тесты в `test_calc.py` / `test_models_and_services.py`.
- Критические сценарии из README и `.claude/agents/qa-testing.md` не удалять и не ломать.

**Запуск тестов:**
```bash
cd backend && pytest                    # все тесты + coverage
cd backend && pytest tests/test_calc.py # только calc
```

---

### 2.4 Guardian (хранитель качества)

**Когда:** после этапа Tester.

**Обязательный выход:**
- **VERDICT:** `APPROVE` или `BLOCK`.
- При BLOCK: перечень конкретных правок (файл, строка или описание).
- Проверки:
  - Lint (flake8/pylint) не ухудшен.
  - Сборка/старт: `docker compose build` / `docker compose up` и при необходимости `npm run build` — без ошибок.
  - Тесты зелёные, новые тесты соответствуют задаче.

**Эскалация:** при повторном BLOCK после правок — привлечение Architect для пересмотра плана.

---

### 2.5 Documenter (документатор)

**Когда:** новая функция, структурное изменение; по решению — после багфикса/рефакторинга.

**Обязательный выход:**
- Список обновлённых документов (README, API-описание, `docs/plans/...`).
- Кратко: что изменилось для пользователя/разработчика.

---

## 3. Схема по типам задач

```
Новая функция:
  Architect (план) → Implementer (код + change-log) → Tester (тесты + отчёт) → Guardian (VERDICT) → Documenter (доки)

Багфикс:
  Debugger (код + change-log) → Tester (тесты/регрессия) → Guardian (VERDICT) → [Documenter при необходимости]

Рефакторинг:
  Analyzer (краткий план) → Refactorer (код) → Tester (регрессия + при необходимости новые тесты) → Guardian → [Documenter]

Структура/архитектура:
  Architect (план) → Implementer/Refactorer → Tester → Guardian → Documenter
```

---

## 4. Минимальный DoD (Definition of Done)

- [ ] Задача классифицирована (новая функция / багфикс / рефакторинг / структура).
- [ ] Пройдены все этапы по схеме для данного типа.
- [ ] Есть план в `docs/plans/` там, где он обязателен.
- [ ] **На все изменения, затрагивающие поведение, есть тесты (backend); для frontend — чек-лист или тесты при наличии фреймворка.**
- [ ] Guardian выдал APPROVE.
- [ ] Lint не ухудшен, сборка и тесты проходят.
- [ ] Документация обновлена там, где требуется по этапу Documenter.
