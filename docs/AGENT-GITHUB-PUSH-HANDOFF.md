# Handoff: как пушить Res-CRM в GitHub (для следующего агента / чата)

Скопируй этот файл целиком или дай агенту ссылку на него в начале нового чата.

---

## 1. Репозиторий и ветка

| Параметр | Значение |
|----------|----------|
| **Репозиторий** | `Istrich/Res-CRM` |
| **URL (SSH)** | `git@github.com:Istrich/Res-CRM.git` |
| **URL (HTTPS)** | `https://github.com/Istrich/Res-CRM.git` |
| **Основная ветка** | `main` |
| **Локальный путь (типичный)** | `/Users/istrich/Service/Res-CRM` |

Проверка перед пушем:

```bash
cd /Users/istrich/Service/Res-CRM
git status
git remote -v
git log -1 --oneline
```

---

## 2. Почему агент в Cursor часто **не может** запушить

- Терминал агента часто в **песочнице**: нет доступа к **SSH-агенту** и к файлам `~/.ssh/*` (приватные ключи).
- Команда `git push` с `git@github.com:...` тогда даёт **`Permission denied (publickey)`** — это нормально для среды агента, а не «сломанный ключ».
- **Публичный** ключ, вставленный в чат, для пуша **не подходит** — нужен **приватный** ключ на машине пользователя (или другой способ ниже).

Итог: «запуши сам» из чата агента **не гарантировано** без одного из вариантов в §3.

---

## 3. Способы, чтобы пуш **реально** прошёл

### Вариант A — пользователь пушит вручную (самый надёжный)

После того как агент сделал `git commit`:

```bash
cd /Users/istrich/Service/Res-CRM
git push origin main
```

Работает, если на Mac уже настроен SSH к GitHub (`ssh -T git@github.com` → успех).

---

### Вариант B — HTTPS + Personal Access Token (PAT)

Подходит, если нужно пушить из среды без SSH.

1. GitHub → **Settings → Developer settings → Personal access tokens**  
   Создать token с правами **`repo`** (для приватного репо) или достаточно для публичного.
2. Переключить remote:

   ```bash
   cd /Users/istrich/Service/Res-CRM
   git remote set-url origin https://github.com/Istrich/Res-CRM.git
   ```

3. При первом `git push` Git запросит логин/пароль: **username** = GitHub username, **password** = **токен** (не пароль от аккаунта).

4. **Никогда** не коммитить токен в репозиторий. Не вставлять токен в чат с AI в открытом виде (лучше завести token в Keychain / credential helper и пушить из обычного терминала пользователя).

Агент может выполнить `git push` **только если** credential уже сохранён на этой машине для HTTPS.

---

### Вариант C — терминал Cursor «без песочницы» / полные права

Если в Cursor есть режим, где команды выполняются **с доступом к SSH агенту пользователя** (как у обычного Terminal.app), тогда:

```bash
cd /Users/istrich/Service/Res-CRM
git push origin main
```

может пройти с тем же ключом, что и у пользователя. Зависит от настроек Cursor/агента — уточнить у пользователя.

---

### Вариант D — GitHub CLI (`gh`)

Если пользователь один раз выполнил `gh auth login`:

```bash
cd /Users/istrich/Service/Res-CRM
gh auth status
git push origin main
```

Снова же: агенту нужен доступ к той же сессии/ключам, что и у пользователя.

---

## 4. Что делать агенту по шагам (рабочий сценарий)

1. Внести изменения, `git add`, `git commit` с понятным сообщением.
2. Запустить тесты (в проекте: backend pytest в Docker с `DATABASE_URL=sqlite:///:memory:`, frontend `npm run build` — см. `.github/workflows/ci.yml`).
3. Попробовать `git push origin main`.
4. Если **publickey** / отказ — **не спорить с пользователем про «ключ из чата»**: написать, что пуш из песочницы недоступен, и дать команду из **варианта A** или попросить включить PAT/полный терминал.

---

## 5. CI на GitHub

В репозитории есть `.github/workflows/ci.yml`:

- backend: pytest + coverage, **flake8** (по всему `app/` могут быть старые предупреждения — проверить перед пушем).
- frontend: `npm ci`, проверка импортов, `npm run build`.

Имеет смысл прогнать то же локально или в Docker перед пушем.

---

## 6. Публичный SSH-ключ (справка)

Ключ с комментарием `Res-CRM GitHub` — это **публичная** половина; он должен быть добавлен в GitHub → **SSH keys**. Пуш идёт с машины, где лежит **соответствующий приватный** ключ (или другой метод из §3).
